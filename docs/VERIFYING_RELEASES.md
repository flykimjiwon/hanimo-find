# Verifying What You Obtained

This document is for a consumer — an auditor, a regulated integrator, anyone who
needs to independently check a copy of Hanimo Find rather than take a
maintainer's word for it. It is the reader's side of the trust boundary.

[RELEASE_GATE.md](../RELEASE_GATE.md) describes the producer-side gate the
maintainer runs before publishing. This document does not repeat that gate. It
describes what a consumer can reproduce and inspect locally from a checkout, and
it is deliberate about what those checks do **not** establish.

Read this together with [SECURITY.md](../SECURITY.md), which defines the runtime
security boundary and the exact meaning of `bundle_sha256`.

## What state the project is in

As of this writing the repository is a source beta. Both workspace members set
`publish = false`, so nothing is on crates.io, and no prebuilt binaries are
attached to any release. Per [SECURITY.md](../SECURITY.md) and
[PUBLICATION_DECISION.md](../PUBLICATION_DECISION.md), no production release,
signed tag, or attached release artifact exists yet. Several verification steps
below therefore describe a procedure you should run against whatever commit you
actually obtained, not a claim that a canonical signed tag is waiting for you.
Where a step depends on something that does not exist yet, this document says so.

## 1. Pin and check out an exact commit

You verify a commit, not a branch. A branch name resolves to a moving target;
only a full commit SHA identifies immutable content.

```sh
git clone https://github.com/flykimjiwon/hanimo-find
cd hanimo-find

# Record the exact commit you are auditing.
git rev-parse HEAD

# Confirm your working tree is clean and matches that commit with no local edits.
git status --porcelain        # expect empty output
git diff --quiet && echo "no unstaged changes"
```

Confirm the commit you checked out is the tip of `origin/main` (or is an ancestor
of it — a branch tip can advance after you clone):

```sh
git fetch origin
git rev-parse origin/main
git merge-base --is-ancestor HEAD origin/main && echo "HEAD is on origin/main history"
```

If you were pointed at a specific tag, check it out by its object and record the
commit it dereferences to:

```sh
git rev-list -n 1 <tag>       # the commit the tag points to
```

Honest limits on this step:

- No release tag is signed at this stage. `git tag --verify` and
  `git verify-commit` will report no signature, because none is attached. Treat
  the commit SHA you recorded, obtained over a channel you trust, as the pin —
  not a cryptographic signature.
- Matching `origin/main` proves your copy agrees with the hosting account's
  current view. It does not by itself prove who controls that account. The
  account identity is an external trust root you must establish separately (see
  section 6).
- The roadmap item that would add append-only, signed provenance
  (Ed25519-signed checkpoints over a Claim ledger) is v0.3 in
  [ROADMAP.md](../ROADMAP.md) and is not implemented here.

## 2. Inspect the CycloneDX SBOM

Committed CycloneDX JSON SBOMs live under [`sbom/`](../sbom/):

- `sbom/hanimo-core.cdx.json`
- `sbom/hanimo-find.cdx.json`

They are CycloneDX BOM format, spec version 1.5, generated by `cargo-cyclonedx`
(pinned to `0.5.9` in [RELEASE_GATE.md](../RELEASE_GATE.md) and the workflow
environment). Each SBOM enumerates the resolved dependency graph as components,
records each crate's version and `purl`, and carries a SHA-256 hash for every
component sourced from the crates.io registry. The workspace's own path member
(`hanimo-core`) appears as a component without a registry hash, because it is
local source rather than a downloaded artifact. The `hanimo-find` SBOM's
top-level metadata component is the `hanimo-find` application package — the
`crates/hanimo-find` crate that builds the `hanimo` binary — recorded with type
`application`.

Basic inspection:

```sh
# Confirm format and spec version.
jq '{format: .bomFormat, spec: .specVersion}' sbom/hanimo-find.cdx.json

# List every component and its version.
jq -r '.components[] | "\(.name) \(.version)"' sbom/hanimo-find.cdx.json | sort

# Confirm registry components carry SHA-256 hashes.
jq '[.components[] | select(.hashes)] | length' sbom/hanimo-find.cdx.json
```

Confirm the SBOM is not stale relative to the lockfile. The producer gate
regenerates the SBOMs from `Cargo.lock` and fails closed on a mismatch (see
[RELEASE_GATE.md](../RELEASE_GATE.md)); you can reproduce that check locally:

```sh
cargo install --locked cargo-cyclonedx --version 0.5.9
./scripts/generate-sboms.sh      # regenerates the committed SBOMs
git diff --quiet -- sbom/ && echo "SBOM matches Cargo.lock"
```

A non-empty `git diff` on `sbom/` means the committed SBOM does not match the
current lockfile and metadata.

What the SBOM does and does not attest:

- It **does** provide a reproducible inventory of the resolved dependency graph
  with per-registry-crate SHA-256 hashes, tied to `Cargo.lock`.
- The committed SBOMs are path-normalized and carry no embedded generation
  timestamp or per-build serial number. That is intentional so the file is
  byte-reproducible from the lockfile; it also means the SBOM is not itself a
  dated or serialized attestation of a particular build event.
- It is **not** signed. It is an inventory, not a provenance signature. Its own
  integrity depends on the commit you verified in section 1, not on any
  cryptographic property of the SBOM file.

## 3. Reproduce the license posture locally

The license posture is enforced by two independent tools, both pinned in
[RELEASE_GATE.md](../RELEASE_GATE.md).

**cargo-deny** ([`deny.toml`](../deny.toml)) enforces advisory, source, and
license policy over the resolved graph. The allowed licenses are exactly
`Apache-2.0`, `Apache-2.0 WITH LLVM-exception`, `MIT`, `Unicode-3.0`, and
`Unlicense`, with an empty `exceptions` list. Sources are restricted to the
crates.io registry (`unknown-registry = "deny"`, `unknown-git = "deny"`,
`allow-git = []`), and wildcard version requirements are denied.

```sh
cargo install --locked cargo-deny --version 0.20.2
cargo deny check
```

**cargo-about** ([`about.toml`](../about.toml),
[`THIRD_PARTY_LICENSES.md`](../THIRD_PARTY_LICENSES.md)) generates the aggregated
third-party notice from the resolved graph. Its `accepted` license list matches
the cargo-deny allow list, and it accounts for build, dev, and transitive
dependencies (none of the `ignore-*` switches are set). Regenerate the notice and
confirm it matches what is committed:

```sh
cargo install --locked cargo-about --version 0.9.1 --features cli
cargo about generate about.hbs > /tmp/THIRD_PARTY_LICENSES.regenerated.md
diff THIRD_PARTY_LICENSES.md /tmp/THIRD_PARTY_LICENSES.regenerated.md \
  && echo "third-party notice matches the resolved graph"
```

A clean `cargo deny check` plus a no-difference regeneration reproduces the
license posture the producer gate asserts. The project's own license is
Apache-2.0, represented by the [`LICENSE`](../LICENSE) text.

## 4. Confirm the MSRV and the locked dependency graph

The minimum supported Rust version is **1.88.0**. It is declared as
`rust-version = "1.88"` in the workspace `Cargo.toml` and stated as `1.88.0` in
the README. Note that [`rust-toolchain.toml`](../rust-toolchain.toml) pins the
`stable` *channel* for day-to-day builds; the MSRV floor is enforced separately
by a dedicated CI job (`.github/workflows/ci.yml`) that runs
`cargo +1.88.0 check --workspace --all-targets --all-features --locked`. You can
reproduce the MSRV check locally:

```sh
rustup toolchain install 1.88.0 --profile minimal
cargo +1.88.0 check --workspace --all-targets --all-features --locked
```

Always build and check with `--locked` so Cargo uses the committed
[`Cargo.lock`](../Cargo.lock) exactly and refuses to silently resolve a
different graph:

```sh
cargo build --locked --release
git diff --quiet -- Cargo.lock && echo "Cargo.lock unchanged by the build"
```

`--locked` errors out rather than editing `Cargo.lock`, so a successful build is
evidence that the committed lockfile is self-consistent with `Cargo.toml`. The
SBOM (section 2) and the license checks (section 3) are all derived from this
same locked graph, so verifying them together checks one coherent dependency set.

## 5. Understand the governance and the admin bypass

Governance is single-maintainer. [`.github/CODEOWNERS`](../.github/CODEOWNERS)
assigns `@flykimjiwon` as the owner of the whole tree and, explicitly, of the
supply-chain surfaces (`/.github/`, `Cargo.lock`, `Cargo.toml`, `deny.toml`,
`about.toml`, `PROVENANCE.md`, `THIRD_PARTY_LICENSES.md`, and `sbom/`). There is
one review authority, not an independent quorum.

Branch and tag protection on the hosting side is a producer-side control. If you
need to rely on it, inspect it yourself against the live repository rather than
trusting a description of it — for example via the GitHub rulesets API — and
record what you observed. Whatever protection exists, weigh it against a
structural fact: on a single-maintainer repository the account owner is also the
administrator, so the owner can change or bypass any ruleset they configured.
That makes host-side tag protection a control the maintainer keeps over
themselves, not an external check a third party enforces. It is not, on its own,
a trust root a consumer can lean on.

Combined with section 1, the honest reading is: host-side protections reduce
accidental or third-party tampering, but they do not cryptographically bind a
release to an identity. Until signed checkpoints exist (v0.3 in
[ROADMAP.md](../ROADMAP.md)), your trust anchor is the commit SHA you pinned over
a channel you already trust.

## 6. What is NOT provided

State this plainly so no consumer over-reads the checks above.

- **No signed release artifacts.** No signed tags, no signed commits, no
  detached signatures, no in-toto/SLSA provenance attestation are published at
  this stage. `git verify-commit` and `git tag --verify` will find nothing to
  verify.
- **No published binaries yet.** No version has been tagged, so no prebuilt
  binaries are attached to a release and nothing is on crates.io
  (`publish = false`). You build from source, which means your toolchain and
  build environment are part of your trust base. When a `v*` tag is released, the
  release workflow (`.github/workflows/release.yml`) attaches a per-target
  archive with a `.sha256` file; verify a download with
  `sha256sum -c <archive>.sha256` before use. Those checksums are unkeyed
  integrity checks (see the next point), not signatures.
- **Digests are unkeyed.** The per-file SHA-256 hashes in the SBOM, the
  `source_sha256` on each evidence block, and `bundle_sha256` are all unkeyed
  integrity checksums. As stated in [SECURITY.md](../SECURITY.md),
  `bundle_sha256` is not a signature, authorization, proof of authorship, or
  trusted timestamp. An actor who rewrites the source, the bundle, and the
  digest together produces a self-consistent artifact that passes every hash
  check. Unkeyed digests detect accidental corruption and inconsistency; they do
  not defend against an actor who controls the inputs.
- **Trust roots are external.** Every check here is relative to a commit and to a
  hosting account. That the commit is authentic, and that the account is
  controlled by whom you think, are facts you must establish out of band. This
  document cannot supply them, and it does not claim to.
- **Local verification is not a truth claim.** As the README notes, verification
  checks current local bytes and bundle self-consistency. It does not establish
  that a cited source is correct or that its author is trustworthy — only that
  the attested bytes are unchanged and match the current source. `hanimo find
  verify` returns exit `0` only when the bundle both re-verifies against live
  sources and passes the recomputed critic rule; see the exit-status contract in
  the [README](../README.md).

## Summary of local checks

| Goal | Command(s) | Reference |
| --- | --- | --- |
| Pin an exact commit | `git rev-parse HEAD`; `git merge-base --is-ancestor HEAD origin/main` | §1 |
| SBOM matches lockfile | `./scripts/generate-sboms.sh`; `git diff --quiet -- sbom/` | §2 |
| License policy | `cargo deny check` | §3, [deny.toml](../deny.toml) |
| Third-party notice | `cargo about generate about.hbs \| diff - THIRD_PARTY_LICENSES.md` | §3, [about.toml](../about.toml) |

| Locked graph | `cargo build --locked --release` | §4, [Cargo.lock](../Cargo.lock) |

These checks are reproducible from the checkout. What they cannot reproduce —
signatures, published binaries, and an external identity for the account — is
listed in section 6 rather than implied away.
